# Maintainers

## Current maintainer

- [@lunaticbugbear](https://github.com/lunaticbugbear) — project owner, release owner, and final reviewer for installer, docs, and security-sensitive changes.

## Maintainer scope

Maintainers are responsible for keeping these project invariants true:

- `install.sh` and `install.ps1` stay aligned where behavior overlaps.
- Uninstall stays conservative by default.
- Browser tooling stays opt-in.
- API bind defaults stay local-only (`127.0.0.1`).
- Runtime files generated by the installer are not committed.
- Hermes state survives rebuilds unless the user explicitly chooses reset/remove-data paths.

## Supported platforms

HADES accepts issues and PRs for:

- Linux with Docker Engine
- macOS with Docker Desktop
- Windows with Docker Desktop and PowerShell
- WSL with Docker integration

Platform-specific changes should state which platform was tested and which platforms are unaffected.

## Review expectations

- Bug fixes should include the exact command or scenario they fix.
- Installer changes should update docs and validation commands when behavior changes.
- Release, security, and default-config changes require maintainer review.
- Changes touching install/uninstall paths should preserve non-interactive mode.
- Release integrity changes require a second look before merge.

## Response expectations

This is a volunteer-maintained OSS project. Maintainers prioritize security reports, installer breakage, and release regressions first. Public issues should include enough environment detail for reproducible triage.

## Governance

- Current maintainer: `@lunaticbugbear`
- Single-maintainer bus-factor is real; automation and docs reduce risk but do not remove it.
- Contributors are welcome for docs, tests, and issue repros.

## Release authority

Only maintainers should create release tags, publish release assets, or change release integrity requirements.

## Emergency patches

If a release is broken or a trust issue ships, pause new releases, fix the source, rerun validation, and cut a follow-up tag with a clear note in the changelog and release notes.
